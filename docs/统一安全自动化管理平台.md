# 统一安全自动化管理平台

## 一、 平台全景拓扑与数据流向

本平台基于**数据编排面与核心原子能力执行面分离**的微服务思想设计，支持 100% 私有化安全部署（可落地于 AWS VPC、GCP 或私有云环境），确保源码、安装包及漏洞资产数据主权完全本地化，满足全球严苛的金融数据隐私保护条例（如 GDPR、PCI-DSS、DORA）。

### 1. 核心架构拓扑


### 2. 控制面高可用与流量削峰加固

为应对成百上千个项目并发执行所带来的瞬时大流量，控制面架构进行了企业级高可用加固：

- **异步削峰机制：** 在 DefectDojo 控制面前置配置 **Celery 分布式任务队列 + Redis 缓存层**。将所有扫描原子节点（如 SonarQube、OWASP ZAP、Nuclei 等）的报告上报由传统的"同步直连写入"重构为"异步队列削峰写入"，彻底避免夜间深扫时并发海量数据导致 Django 进程崩溃或 PostgreSQL I/O 锁死。
- **兜底拉取模式（Pull）：** 在 RESTful API 异步上报（Push）的基础上，平台为静态白盒和基础架构审计增加了主动拉取（Pull）备用机制。系统通过定时 Cron Job 触发控制面主动调用各原子工具的 API 抓取漏报数据，全面防范因构建机网络抖动或丢包导致的审计数据丢失。

## 二、 核心选型决策依据与技术指标

### 1. 编排大脑：OWASP DefectDojo

- **定位：** 全局漏洞资产流水账与生命周期管理中心。
- **技术优势：** 原生支持主流开源与商业扫描器的数据解析导入，具备基于 `CWE编号 + URI路径 + 参数散列` 的去重引擎。其输出报告具备全球监管公信力。

### 2. 静态白盒与依赖审计：SonarQube & OSV-Scanner

- **定位：** 编译期源码安全与开源软件供应链（SCA）防线。
- **技术优势：** 针对传统 `pnpm audit` 在高并发 CI/CD 流水线中因频繁请求外部网络导致的网络抖动、超时及构建中断问题，平台全面将依赖审计升级为 **OSV-Scanner 离线比对** 机制。通过本地定期同步的开源漏洞库（Open Source Vulnerabilities），实现无网确定性防御，杜绝供应链投毒。

### 3. H5 动态黑盒：OWASP ZAP + Playwright (Python)

- **定位：** 模拟黑客视角，通过流量被动代理，深度挖掘 H5 单页应用的业务逻辑缺陷及 OWASP Top 10 运行时漏洞。
- **技术优势：** Playwright 负责使用标准移动端浏览器 UA 跨越海外复杂的 OAuth 2.0 登录墙并驱动核心业务流；OWASP ZAP 作为黑盒被动代理拦截流量并进行漏洞深度挖掘。
- **可观测性（TraceID 联动）：** 执行机在发起探测时，必须强制注入标准 **OpenTelemetry / X-B3-TraceId** 等分布式链路追踪标头。确保所有自动化渗透测试流量在后端微服务群中的每一次 RPC 调用、每一次 Redis 读写及 SQL 执行轨迹均全量可留存、可监控（联动 Grafana/Jaeger）。使研发人员在面对黑盒误报或服务状态异常时，能够**秒级复现黑客攻击上下文**，彻底消除研发与安全团队的沟通扯皮。

### 4. 移动端逆向：MobSF (Mobile Security Framework)

- **定位：** 针对海外应用市场的移动端合规审计与逆向引擎。负责 APK/IPA 反编译、本地隐私合规、硬编码密钥及 SSL Pinning 绕过风险审计。

### 5. 后端 API 与基础架构扫描器：Nuclei

- **定位：** 基于 YAML 模板的高并发、精准漏洞秒杀引擎。专门应对海外最新的 1day/0day 漏洞。在流水线中通过配置不同的 Tag，可在几十秒内对后端全线微服务网关进行撞库式漏洞筛查。

## 四、 金融级"防资损/防脏数据"断路器设计

在自动化黑盒渗透中，动态工具会并发大量带有攻击载荷的请求。系统必须在网关层、应用框架层、中间件层全面落实"全链路流量染色与数据沙箱隔离"工程契约：

```text
[ 自动化执行机流量 ] ──> 发出请求携带 Header: X-SecOps-Simulation & X-SecOps-Sign (HMAC) & X-B3-TraceId
                              │
                              ▼
           【 测试环境边缘网关层 (AWS ALB / Nginx / Gateway) 】
                              │ (前端标头强制清洗 Header Sanitization)
               ┌──────────────┴──────────────┐
               │ 校验 1: 是否包含染色 Header? │
               │ 校验 2: HMAC 动态时间戳验签   │
               │         (时间窗口限制 ±5min)  │
               │ 校验 3: 来源 IP 是否在白名单?│
               └──────────────┬──────────────┘
                              │
               ┌──────────────┴──────────────┐
         [ 是: 进入沙箱路由 ]          [ 否: 走常规流程 ]
               │                             │
    ┌──────────┴──────────┐                  ▼
    ▼                     ▼            【 真实测试数据库 】
【 消息发送接口 】   【 钱包充提接口 】
(网关直接拦截并返回) (路由至内存 Mock 桩)
(200 OK 假日志)     (资金不真正入库)
    │                     │
    └──────────┬──────────┘
               ▼
   【 中间件防污染：影子区隔离 】
   - Redis: 自动追加入前缀 secops:
   - Kafka/RabbitMQ: 自动路由至影子队列
```

### 1. 全局流量染色与动态签名

Playwright 驱动及 Nuclei探测时，强制在请求中注入全局自定义 Header：`X-SecOps-Simulation: True`。同时，为了彻底杜绝染色 Header 被恶意攻击者截获、重放以绕过金融扣款逻辑，系统强制启用 **HMAC 动态时间戳签名机制**。

- **执行机生成：** 请求发起时，基于当前时间戳 $T$ 与预设的安全盐值 $Salt$，通过 HMAC-SHA256 算法生成动态数字签名，并限定在 **$\pm 5$ 分钟时间窗口**内，超时直接判定为重放攻击：

$$\text{X-SecOps-Sign: } \text{HMAC-SHA256}(Salt, T)$$

### 2. 边缘网关层：防标头伪造与三重鉴权

测试环境边缘网关（如 Nginx / AWS ALB）在执行三重校验前，必须显式进行 **前端标头强制清洗（Header Sanitization）**，清理掉所有来自外部公网请求中携带的 `X-SecOps-` 前缀标头。严格防止外部恶意攻击者利用前端反向代理在 HTTP Header 处理上的隐蔽缺陷（如 HTTP 标头走私 HTTP Header Smuggling、大小写绕过等），伪造内网执行机身份来白嫖 Mock 桩。通过清洗后，网关对内网流量执行强力校验：签名合法性校验、时间窗口校验、执行机固定内网 IP 白名单校验。

### 3. 计算与中间件数据的全方位隔离（影子区设计）

夜间动态黑盒（ZAP/Playwright）强攻压测时，带有攻击 Payload 的请求虽未写入核心 DB，但极易被写进公共缓存（Redis）或异步消息队列（Kafka/RabbitMQ），导致正常测试用户的排队序列或配置被污染破坏。为此，后端框架必须进行中间件脱钩：

- **消息/通知接口：** 网关识别后直接拦截，不调用海外真实的短信或邮件网关，直接在本地打印日志并返回 `200 OK`。
- **金融扣款/充值接口：** 自动重定向到后端桩代码（Mock Stub），数据仅在内存中打转。
- **中间件影子隔离：** 后端应用框架在识别到染色状态后，**对读写 Redis 的 Key 自动加上 `secops:` 前缀（命名空间隔离）**，并对发送的 MQ 消息**自动路由至独立的测试影子队列或死信队列**，实现真正金融级的数据与中间件全链路物理隔离。

## 五、 全自动化"无痛"接入规范（研发脚手架）

### 1. 基于合规流水线（Compliance Pipelines）的硬卡点

为了防止成百上千个项目由于本地配置被误删或恶意篡改而逃避审计，平台改变了传统单纯依靠项目本地 `.gitlab-ci.yml` 自主 `include` 的柔性规范，升级为利用 **GitLab 强制合规流水线（Compliance Pipelines）** 机制。

由安全中台在 GitLab 组（Group）级别将中央安全流水线模板进行**强制置顶注入**。项目本地配置只能定义变量，无法通过删除 `include` 或篡改 CI 逻辑逃避审计。

### 2. 上报凭证与项目 ID 越权校验

安全大脑（DefectDojo）在接收上报报告时，后端 API 必须进行二次越权校验：强制比对当前 Runner 的凭证令牌、当前 Git 仓库路径与传入的 `SECURITY_PRODUCT_ID` 是否严格一致，防止研发人员为了逃避自身项目漏洞追责，采用"李代桃僵"的方式篡改 ID 上报到其他干净的项目中。

```yaml
# 项目本地无需编写任何 include 逻辑，仅需定义合规声明变量
variables:
  SECURITY_PRODUCT_ID: "105" # 必须与 Git 仓库路径严格绑定，中台强校验
  TARGET_SCAN_URL: "https://staging-wallet.yourdomain.com" # 动态黑盒目标
```

## 六、 核心技术风险、注意事项与安全加固方案

### 1. 扫描服务器的"反向 RCE 风险"（平台自身安全防护）

- **风险：** 在使用 MobSF 反编译或使用 Nuclei/ZAP 强攻高危流量时，扫描器底层的解析组件可能被恶意 Payload 反向攻破，威胁测试基础设施。
- **避坑指南：** 所有扫描原子节点必须使用 Docker 容器化非特权模式死隔离部署。安全中台所在的网段在公司 VPC 中必须设置为"只进不出"的隔离网段。

### 2. 国际应用市场的"加固壳"冲突

- **风险：** 金融 App 上线前必须通过防篡改工具加固，导致 MobSF 失效。
- **避坑指南：** 必须严格卡死流水线执行顺序（如阶段 C 所示）：源码编译 $\rightarrow$ 产出未加固的 Staging 包 $\rightarrow$ 校验 SHA-256 哈希链并推送至 MobSF $\rightarrow$ 扫描通过后触发海外混淆加固 $\rightarrow$ 产出最终发布包。

### 3. 柔性红线熔断机制与高权限紧急逃生通道（Bypass）

- **风险：** 安全卡点过死会严重拖慢海外业务交付速度，导致研发集体反弹。同时，必须预防安全大脑自身宕机或网络故障时，导致全公司紧急 Hotfix 发版被强行卡死的极端灾难。
- **避坑指南：** 依据国际 CVSS 评分在 DefectDojo 中配置精细化熔断策略（Critical/High 强制阻断，Medium 触发 Slack 报警并给 3~7 天宽限期）。针对控制面宕机等极端场景，流水线内置**分布式紧急逃生通道**：
- 在工程上允许通过组级别受控的 `SECURITY_BYPASS_TOKEN` 强行跳过安全扫描放行编译。但此放行属于最高合规红线，系统将**实时触发 PagerDuty/Slack 顶格报警抄送 CTO 团队**，且该项目将自动被打上"非安全合规发布"标签，限期 24 小时内必须由安全委员会进行战术级人工复核与补扫闭环。

### 4. 安全大脑（DefectDojo）自身的安全壁垒

- **风险：** DefectDojo 集中存储了全线项目的源码缺陷和高危漏洞报告，是典型的攻击靶心。
- **避坑指南：**
  - **强认证机制：** 控制面后台必须强制开启 **多因素认证（MFA）**，严禁仅依靠单一密码登录，且登录日志实时同步至监控总线。
  - **存储加密策略：** 数据库（PostgreSQL）涉及项目漏洞细节、敏感资产凭证的字段，必须在底层启用**透明数据加密（TDE）**与**落盘加密（Encryption at Rest）**，确保即使虚拟磁盘或数据库备份泄漏，黑客也无法离线还原数据。

### 5. 极端场景下的分布式逃生与硬审计追溯

- **风险：** 安全大脑（DefectDojo）或安全网段若遭遇不可抗力宕机、抖动，会导致全公司正常的业务紧急 Hotfix 发版被强行卡死在流水线，引发次生资损。
- **避坑指南：** 流水线内置高权限紧急逃生通道。通过配置组级别受控的 `SECURITY_BYPASS_TOKEN` 可强行放行编译，但此操作属于最高合规红线，系统将实时向安全委员会及 CTO 团队抄送 PagerDuty 强审计日志，放行项目将自动被打上"非安全合规发布"标签，限期 24 小时内人工复核补扫。

### 6. 流量染色的全链路分布式追踪与中间件沙箱

- **风险：** 夜间动态黑盒（ZAP/Playwright）强攻压测时，产生的漏洞载荷可能导致微服务内部发生隐蔽的内存泄漏或逻辑死循环，且数据易污染公共缓存与消息队列。
- **避坑指南：** 执行机探测时网关层强制注入标准 `X-B3-TraceId` 链路追踪上下文；同时后端微服务需识别染色状态，对 Redis/MQ 的读写自动路由至带有 `secops:` 前缀的影子区。确保所有安全测试产生的流量全链路可留存、可监控，做到"漏洞发现即上下文复现"。

## 七、 第三方专业机构人工测试边界与介入节点

自动化平台主要负责阻断常规、高频的已知风险。基于金融业务对"业务逻辑越权"、"条件竞争/并发资损漏洞"以及"国际审计法律效力"的硬性要求，必须在以下关键节点引入独立第三方测试机构进行人工渗透测试（Penetration Testing）：

1. **核心支付与鉴权大改动前夕：** 凡涉及核心钱包逻辑、转账体系重构、海外第三方支付网关迁移，以及统一登录鉴权系统大版本变更时。
2. **海外法律合规审计期：** 满足 **PCI-DSS（支付卡行业数据安全标准）年度核查**、GDPR 或 DORA 经营韧性合规核查等硬性合规时间表，必须使用有国际资质的第三方报告提供法律背书。
3. **大厂 B 端商业准入门槛：** 对接海外大牌 B 端客户、跨国银行或主流金融机构，需提供近 6 个月内的独立第三方灰盒渗透测试报告作为供应链准入准许证。
4. **重大安全事件响应后：** 系统若遭受外部恶意攻击、撞库或数据泄漏，在内部团队完成应急响应与补丁修复后，引入无利益冲突的第三方进行战术级复核。

- **第一阶段：中台骨架搭建与已有 SonarQube 同步**
