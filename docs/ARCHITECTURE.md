## 设计理念

### 核心设计原则

1. **控制面与数据面分离**
   - 控制面负责管理、编排、展示，运行在安全区域
   - 数据面负责执行扫描、流量处理，可水平扩展
   - 两者通过标准 API 解耦，支持独立演进

2. **生产无侵入**
   - 安全测试不影响真实业务数据
   - 流量染色 + 影子模式，实现"零影响"测试
   - 染色请求与真实请求物理隔离

3. **智能化运营**
   - 多源数据统一接入
   - 智能去重减少噪音
   - 自动化程度逐步提升

4. **企业级安全合规**
   - 全链路审计追踪
   - 细粒度权限控制
   - 符合等保 2.0 要求

## 流量染色机制

流量染色是平台的核心创新点，通过对安全测试请求进行特殊标记，实现生产环境的无侵入测试。

### 核心概念

#### 什么是流量染色？

为安全测试的 HTTP 请求添加一组特殊的请求头（染色标识），业务系统识别这些标识后，将请求路由到影子资源（影子库、影子队列等），确保测试数据不污染真实业务数据。

#### 染色标识组成

| Header | 说明 |
|--------|------|
| `X-SecOps-Simulation` | 模拟标识，值为 `True` 表示是染色请求 |
| `X-SecOps-Sign` | HMAC-SHA256 签名，防止伪造 |
| `X-SecOps-Timestamp` | 请求时间戳，防止重放 |
| `X-B3-TraceId` | （可选）链路追踪 ID |

### 工作流程

```
 安全测试端                     业务系统                     影子资源
     │                            │                            │
     │  1. 生成染色请求            │                            │
     │     - 生成时间戳            │                            │
     │     - 计算 HMAC 签名        │                            │
     │     - 设置染色 Header       │                            │
     ├───────────────────────────►│                            │
     │                            │  2. 校验染色标识            │
     │                            │     - 检查 Simulation Header│
     │                            │     - 验证签名有效性        │
     │                            │     - 检查时间窗口          │
     │                            │                            │
     │                            │  3. 路由决策                │
     │                            │     是染色请求？            │
     │                            │       ├─ 是 → 影子资源      │
     │                            │       └─ 否 → 真实资源      │
     │                            ├───────────────────────────►│
     │                            │                            │
     │                            │  4. 执行业务逻辑            │
     │                            │     （使用影子数据）        │
     │                            │                            │
     │  5. 返回结果               │                            │
     │◄───────────────────────────┤◄───────────────────────────┤
     │                            │                            │
```

### 安全机制

#### 1. HMAC 签名防伪造

- 使用 SHA-256 HMAC 算法
- 密钥仅安全团队和业务系统知晓
- 签名内容 = 时间戳 + 盐值

#### 2. 时间窗口防重放

- 默认时间窗口：300 秒（5 分钟）
- 请求时间戳超出窗口则拒绝
- 可根据场景调整窗口大小

#### 3. IP 白名单（可选）

- 可配置仅允许特定 IP 段的染色请求
- 进一步缩小攻击面

#### 4. 常量时间比较

- 使用 `timingSafeEqual` 进行签名比较
- 防止时序攻击

### 影子资源命名规范

| 资源类型 | 前缀/后缀 | 示例 |
|----------|----------|------|
| Redis Key | `secops:shadow:` 前缀 | `secops:shadow:user:123` |
| MQ 队列 | `-shadow` 后缀 | `order-queue-shadow` |
| 数据库 Schema | `shadow_` 前缀 | `shadow_orders` |

### 流量染色代码示例

```typescript
import { TrafficDye } from '@secops/traffic-dye';

const dye = new TrafficDye({
  salt: 'your-secret-salt',
  timeWindowSeconds: 300,
});

// 生成染色请求头
const headers = dye.generateHeaders('trace-id-123');
// {
//   'X-SecOps-Simulation': 'True',
//   'X-SecOps-Sign': 'abc123...',
//   'X-SecOps-Timestamp': '1717225600',
//   'X-B3-TraceId': 'trace-id-123'
// }

// 验证请求
const result = dye.verify(req.headers, req.ip);
if (result.valid) {
  // 路由到影子资源
  const shadowKey = dye.getShadowRedisKey('user:123');
}
```

## 去重引擎设计

智能去重引擎是提升安全运营效率的核心组件，大幅降低重复漏洞的干扰。

### 为什么需要去重？

在实际安全运营中：
- 多个扫描工具可能报出同一个漏洞
- 同一次扫描中同一类问题可能出现多次
- 不同分支/版本的相同漏洞重复上报
- 误报反复出现，占用大量人工复核时间

### 去重哈希算法

#### 核心原理

对漏洞的关键特征进行归一化处理，然后计算哈希值，相同哈希的漏洞判定为重复。

#### 特征维度

| 特征 | 说明 | 归一化方式 |
|------|------|-----------|
| CWE/CVE | 漏洞类型标识 | 大写、去空格 |
| 文件路径 | 漏洞所在文件 | 原始路径 |
| 代码行号 | 漏洞起始行 | 数字 |
| 位置 | URL / API 端点 | 原始字符串 |
| 漏洞标题 | 漏洞名称 | 小写、去空格 |
| 参数 | 请求参数 | 按键排序后拼接 |

#### 去重类型

**1. SAST 去重**

```
哈希键 = CWE + 文件路径 + 起始行号
```

**2. DAST 去重**

```
哈希键 = CWE + URL路径 + 参数（排序后）
```

**3. SCA 去重**

```
哈希键 = CVE + 包名 + 版本号
```

### 去重流程

```
新漏洞进入
    │
    ▼
提取关键特征
    │
    ▼
特征归一化
    ├─ CWE/CVE → 大写、trim
    ├─ 文件路径 → trim
    ├─ 参数 → key 小写、按字母排序
    └─ 标题 → 小写、trim
    │
    ▼
计算 MD5 哈希
    │
    ▼
查询已有漏洞
    │
    ├─ 哈希已存在 → 标记为重复，关联到原漏洞
    └─ 哈希不存在 → 新漏洞，入库
```

### 去重效果

- **跨工具去重**：不同扫描器的同一漏洞合并展示
- **跨版本去重**：不同代码版本的相同问题只保留最新
- **跨扫描去重**：多次扫描的重复结果自动合并

### 去重查询 API

平台提供去重哈希的查询接口，可快速定位同一漏洞在不同项目/扫描中的出现情况。
